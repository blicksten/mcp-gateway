# ADR-0005: Claude Code Integration — Dual-mode gateway + Plugin packaging + Webview patch (Alt-E native reconnect)

**Status:** Accepted
**Date:** 2026-04-22
**Deciders:** Porfiry (architect) + multiple PAL consultation rounds
**Version:** v1.6.0

## Context

Three prior-audit findings drove this phase:

1. **HIGH — bootstrap gap between Claude Code and mcp-gateway.** Claude Code
   could not auto-discover the gateway's backends; users had to
   hand-configure `.mcp.json` per project.
2. **MEDIUM — tools/list caching (Issue #13646).** Claude Code caches
   `tools/list` aggressively; adding a new backend on a running gateway
   did not invalidate the cache. The user had to restart Claude Code to
   see new tools.
3. **MEDIUM — `.claude/commands/*.md` semantic confusion.** Operators
   and AI alike routinely treated slash-command prompt templates as MCP
   server registrations. Both exist under `.claude/`, both are markdown,
   but they register into completely different subsystems.

## Decision

We ship a **hybrid** approach combining four layers, each solving one
slice of the problem:

### Layer 1 — Dual-mode gateway (Phase 16.1)

The gateway serves TWO MCP surfaces from a single daemon:

- `/mcp` — aggregate: all backends exposed via namespaced tool names.
- `/mcp/{backend}` — per-backend: one MCP server per backend, unnamespaced.

The dual surface lets a Claude Code plugin register separate `plugin.json`
entries for each backend (so tool names are stable per plugin) while
keeping the aggregate endpoint available for clients that prefer a single
server.

### Layer 2 — Claude Code Plugin packaging (Phase 16.2)

We ship `installer/plugin/` as an installable Claude Code plugin via the
`claude plugin install mcp-gateway@mcp-gateway-local` marketplace path.
The plugin's `.mcp.json` is regenerated from the gateway's live backend
list on every REST mutation (add/remove/disable) via
`plugin.Regenerator`. The gateway OWNS the file; hand-edits are
overwritten on the next regen (REVIEW-16 L-05).

### Layer 3 — Webview patch using native `reconnectMcpServer` (Alt-E)

**Alt-E is the key architectural pivot.** The original PLAN-16 §16.4
design was to invoke `executeCommand("reload-plugins")` from a patched
webview bundle. The spike (`docs/spikes/2026-04-20-reload-plugins-probe.md`)
proved that:

- The string literal `reload-plugins` does NOT exist in the Claude Code
  webview bundle.
- A native method `session.reconnectMcpServer(name)` IS exposed on the
  webview's React fiber tree at depth 2 (same neighborhood as
  `session.setModel` which `porfiry-taskbar.js` already walks).
- The `/mcp` panel's "Reconnect" button in Claude Code 2.1.114 calls
  exactly this method.

Under Alt-E, our patch (`installer/patches/porfiry-mcp.js`) walks the
fiber tree, captures a reference to `session.reconnectMcpServer`, and
calls it directly when the gateway enqueues a `reconnect` action. No
`executeCommand`, no extension.js patch, no toggle-hack.

### Layer 4 — Semantic disclaimer on slash commands (Phase 16.9.3)

Every auto-generated `.claude/commands/*.md` file now starts with:

```
<!-- AUTO-GENERATED by mcp-gateway extension. DO NOT EDIT — will be overwritten. -->
<!-- NOTE: This is a slash-command prompt template, NOT an MCP server registration. -->
<!-- MCP servers are registered via the mcp-gateway plugin (see README §Connecting Claude Code). -->
```

The disclaimer is pinned by a regression test in
`slash-command-generator.test.ts` (all three lines must appear as the
first three lines of every generated file).

## Rejected alternatives

Detailed options-analysis is preserved in the spike report. Summary of
what we did NOT do and why:

- **HTTP reverse proxy.** MCP is stateful (initialize → session → tools/list
  → call). A plain HTTP reverse proxy cannot transparently bridge two
  independent MCP sessions; we would reimplement half the SDK.
- **Suppress aggregate tools when plugin loaded.** Produces a
  disappearing-tools UX when the user toggles the plugin off. Violates
  the "what you see is what you get" property; rejected.
- **SIGHUP wrapper on the gateway.** Invasive shell-rc edit to trap
  signals and reload state in-process. Rejected for portability (Windows
  has no SIGHUP) and operator-fragility.
- **Socket injection.** Unofficial primitive; depends on Claude Code's
  internal IPC surface which has no stability guarantees. Rejected for
  the same reason we rejected `executeCommand("reload-plugins")`.
- **`executeCommand("reload-plugins")` fiber walk.** Proven by the spike
  to not exist in the webview bundle. The three separate passes
  documented in the spike report (static grep → DOM inspection → live
  probe) all confirmed the absence.
- **`extension.js` dispatcher patch.** Superseded by Alt-E. Alt-E uses
  `session.reconnectMcpServer` — a method Claude Code itself calls from
  its own `/mcp` panel "Reconnect" button. Patching extension.js would
  have added an extra failure surface for no additional capability.
- **`set_plugin_enabled` toggle trick.** Simulate a plugin reload by
  disabling + re-enabling it. Produces a user-visible disable flicker in
  the `/mcp` panel; only works at plugin granularity (we want per-backend
  reload). Rejected for UX.

## Consequences

**Fragility:** The webview patch still depends on the shape of Claude
Code's internal React fiber tree. A major CC refactor could break the
fiber walk. This risk is mitigated in three ways:

1. **Native primitive.** Alt-E calls the SAME method Claude Code's own
   `/mcp` panel calls. Internal API stability is significantly higher
   than arbitrary internal commands — if Claude Code moves
   `reconnectMcpServer`, its own UI breaks first, forcing a fix.
2. **Supported-versions map.** `configs/supported_claude_code_versions.json`
   tracks `alt_e_verified_versions`. The dashboard surfaces Mode C (yellow
   advisory) when the live CC version is not in the verified set.
3. **Aggregate fallback.** The `/mcp` aggregate endpoint keeps working
   regardless of patch state. Users who decline the patch get manual
   reload via the `/mcp` panel "Reconnect" button — still native, still
   supported, just opt-in manual instead of automatic.

**Opt-in rails:** The webview patch is explicitly opt-in via the
dashboard checkbox or `mcp-ctl install-claude-code` (default) /
`--no-patch` (opt-out). The operator who chooses it accepts that the
maintainer release has been reviewed for what it modifies.

**Maintenance cost:** A new CC version requires a spike pass to confirm
`alt_e_verified_versions` still holds, followed by a compat-matrix
update. The matrix is versioned in-repo (`configs/supported_claude_code_versions.json`)
and served via `GET /api/v1/claude-code/compat-matrix` so the dashboard
can flag drift in real time.
